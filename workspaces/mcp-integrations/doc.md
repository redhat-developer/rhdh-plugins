# Red Hat \- MCP Development Requirements for Products and Services

## Executive Summary

This document provides minimal additional guidance for engineering teams building [Model Context Protocol (MCP)](https://modelcontextprotocol.io/specification/2025-03-26) servers and integrating MCP servers into agentic applications for use in customer-facing Red Hat products and services. It supplements, not replaces, the official specification.

The document covers:

- **MCP Client/Host Application Requirements**: Security and control requirements for applications integrating MCP servers
- **MCP Server Evolution & Testing**: Guidelines for evolving server APIs and validating against multiple LLM providers
- **MCP Server Best Practices**: Implementation requirements organized as REQUIRED (must-have for production) and RECOMMENDED (should-have practices), plus guidance on code execution patterns

**For existing servers**: If your server is out of sync with these requirements, prioritize engineering work to align with REQUIRED items first, then RECOMMENDED.

**Feedback appreciated**: This document is intended to provide a baseline set of requirements to guide Red Hat product and services building MCP servers. If there is a requirement in this document which you feel is not appropriate for your particular product or service scenario, please reach out with a suggested change or modification. Feedback is warmly welcomed.

| Model                     | Host Site               | Communication | Key Technical Constraint                                                    |
| ------------------------- | ----------------------- | ------------- | --------------------------------------------------------------------------- |
| **Local (Binary)**        | User Desktop            | `stdio`       | Requires local installation; no network exposure needed for the MCP itself. |
| **Self-Hosted (Service)** | Local, Private network  | `SSE/HTTP`    | Managed by the user/team; runs as a persistent network service.             |
| **Customer infra**        | Customer infrastructure | `SSE/HTTP`    | Managed by the customer, we can provide best practices but no guarantees    |
| **Managed PaaS**          | IT Supported infra      | `SSE/HTTP`    | Fully managed; accessed via public or peered internet.                      |

### Axis 4: Data classification

[Reference](https://source.redhat.com/departments/strategy_and_operations/it/it_information_security/wiki/red_hat_data_classifications)

- **RH-Internal** \-Information that is limited to Red Hat employees as well as contractors and vendors engaged to manage this information and covered by NDA or confidentiality agreement.
- **RH-Restricted** \- Information intended for use by a person or group who has a demonstrated need to know. Recipients of this information are generally bound by confidentiality or non-disclosure agreements or the sender has sought explicit legal approval before providing this information.
- **RH-Public** \- Information that has been explicitly approved by Red Hat management for release to the public
- **RH-Restricted(+PII)** \- RH-Restricted information (see row above) that includes PII. PII stands for information that is “personal data” or “personally identifiable information” (PII) or a similar term used to describe information relating to an identified or identifiable (by virtue of the information) natural person.

## MCP Client/Host Application Requirements

Requirements for applications integrating MCP servers (agentic systems, AI assistants, automation platforms).

| Requirement                             | Responsibility          | Description                                                 | Implementation                                                                                                                                                              | Rationale                                                                             |
| :-------------------------------------- | :---------------------- | :---------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------ |
| **Least Privilege \- Minimal Tool Set** | Client/Host only        | Enable only the minimal set of MCP servers and tools needed | • Advertise only necessary servers and tools • Filter to task-specific subsets                                                                                              | Reduces attack surface and improves tool selection accuracy, token use                |
| **Prompt Injection Protection**         | Shared (Client primary) | Implement defenses against prompt injection attacks         | • Inspect and sanitize tool responses • Harden system prompts against embedded commands • Use guardrails to detect jailbreaking attempts in prompt content and tool output. | User content can manipulate LLM to bypass access controls or leak data                |
| **Human-in-the-Loop Controls**          | Client/Host only        | Implement approval mechanisms for sensitive operations      | • Intercept high-risk tool calls for user approval                                                                                                                          | There SHOULD always be a human in the loop with the ability to deny tool invocations. |

## MCP Server Evolution & Testing

| Recommendation                           | Description                                                                                              | Implementation                                                                                                                                                                   | Rationale                                                                            |
| :--------------------------------------- | :------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------- |
| **Measurement-Driven Evolution**         | Prioritize measured model performance over backward compatibility when evolving servers                  | • Establish baseline eval metrics • Measure impact on model behavior (tool selection accuracy, task completion) • Make breaking changes if a significant improvement warrants it | MCP is about LLM-tool interaction, which is rapidly evolving. Optimize for outcomes. |
| **Multi-Model Testing**                  | Test against multiple LLM providers if supporting broad adoption                                         | Test with different models to identify compatibility issues                                                                                                                      | Different models interpret tool descriptions differently                             |
| **Future-Proof Architecture Assessment** | Create as few static tools as possible while considering potential shifts toward code execution patterns | • Assess if users could accomplish via API/SDK \+ code execution • Design minimal, composable tools rather than comprehensive catalogs                                           | Minimizes maintenance burden and prepares for potential paradigm shifts              |

## MCP Server Best Practices

### REQUIRED

All items in this section MUST be implemented before production deployment. Review each requirement and verify your server meets all criteria.

| Requirement                                                                                               | Description                                                                                                                                                                                      | Implementation                                                                                                                                                                                                                                                                                                                                                                                                                       | Rationale                                                                                                                                   |
| :-------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------ |
| **Read-Only Mode by Default**                                                                             | All MCP servers MUST support read-only mode and deploy in this mode by default                                                                                                                   | • Write operations only advertised when explicitly enabled                                                                                                                                                                                                                                                                                                                                                                           | Defense-in-depth security; minimizes blast radius of misconfiguration or compromise                                                         |
| **Sensitive Data Redaction TODO: LEGAL input is needed on this row. Existing language is a placeholder.** | Servers MUST be deployed with a mechanism to configure redaction of sensitive information in tool outputs, either natively or via an MCP Gateway with guardrail support. Logs must be sanitized. | • **Native Implementation**: Support patterns for API keys, tokens, passwords, secrets, emails, and IP addresses • **Alternative**: Deploy behind an MCP Gateway that provides redaction/guardrail capabilities                                                                                                                                                                                                                      | Sensitive data MUST NOT be included in LLM conversation context absent explicit opt-in during Agentic application or MCP server deployment. |
| **Auth(z)**                                                                                               | Servers MUST implement auth(z) mechanisms appropriate to their transport                                                                                                                         | Follow best practices from MCP [Authorization](https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization) spec for your transport                                                                                                                                                                                                                                                                                 | Match authentication to deployment context                                                                                                  |
| **Error Handling \- No Information Leakage**                                                              | Servers MUST NOT expose sensitive system information in error messages                                                                                                                           | **Prohibited**: Internal file paths, stack traces with system details, credentials, database schema **Allowed**: Generic error codes, user-actionable messages, sanitized context                                                                                                                                                                                                                                                    | Prevents reconnaissance and information disclosure                                                                                          |
| **Mandatory Timeouts**                                                                                    | All long-running operations MUST have configurable timeout limits                                                                                                                                | • Default timeout values for all operation types • User-configurable timeout overrides (within max bounds) • Graceful timeout handling with partial results if available • Clear timeout error messages with guidance                                                                                                                                                                                                                | Prevents resource exhaustion and hangs                                                                                                      |
| **Schema Definition & Validation**                                                                        | All tools MUST have complete JSON schemas defining inputs and outputs, and all parameters MUST be validated against schemas before execution                                                     | • Reject requests that don't match schema • Return clear validation error messages                                                                                                                                                                                                                                                                                                                                                   | Ensures data integrity, prevents malformed or malicious inputs, enables automatic validation and client code generation                     |
| **MCP Protocol Compliance**                                                                               | Servers MUST strictly comply with the MCP specification, including JSON-RPC 2.0, initialization, capability negotiation, and versioning                                                          | • Build servers with official or well maintained MCP SDK frameworks                                                                                                                                                                                                                                                                                                                                                                  | Ensures interoperability with MCP clients and allows safe evolution of server capabilities                                                  |
| **Tool Documentation**                                                                                    | All tools MUST have comprehensive schemas and descriptions with LLM-friendly usage hints                                                                                                         | • Describe expected input patterns and common use cases                                                                                                                                                                                                                                                                                                                                                                              | Improves LLM tool selection accuracy, reduces errors                                                                                        |
| **Server Documentation**                                                                                  | MUST include comprehensive documentation with purpose, installation, configuration, usage examples, and security considerations                                                                  | • Purpose and use cases • Required permissions and credentials • Security best practices for deployment                                                                                                                                                                                                                                                                                                                              | Essential for developers onboarding, adoption, secure deployment, and risk assessment                                                       |
| **Operational Readiness**                                                                                 | Servers MUST implement health checks, graceful shutdown, structured logging, and resource cleanup                                                                                                | • **Health Checks**: Endpoint or tool that returns server health and connectivity to dependencies • **Graceful Shutdown**: Handle shutdown signals, close connections, flush logs, and release resources • **Logging**: Structured logging for tool invocations, errors, authentication events (without sensitive data) • **Resource Cleanup**: Automatic cleanup of temporary resources on error, timeout, completion, and shutdown | Enables monitoring and alerting, prevents data loss and resource leaks, essential for debugging, security auditing, and compliance          |
| **Model-Based Evaluation Testing**                                                                        | MCP servers MUST be tested against an actual LLM using an evaluation framework before production deployment                                                                                      | • Create eval dataset with representative user queries/tasks • Measure: tool selection accuracy, parameter correctness, task completion rate, error handling • Establish baseline metrics and regression thresholds • Run evals in CI/CD pipeline                                                                                                                                                                                    | Static tests don't reveal how LLMs actually interact with tools. Real-world model behavior often surprises developers                       |
| **Stateless & Idempotent Tool Design**                                                                    | Tools MUST be stateless and idempotent                                                                                                                                                           | • Each invocation is independent with explicit context • Same operation executed multiple times produces the same result                                                                                                                                                                                                                                                                                                             | LLMs struggle with multi-step stateful protocols. Enables safe retries and better LLM compatibility                                         |

### RECOMMENDED

These practices significantly improve server quality, reliability, and maintainability. Review each item and implement as engineering capacity allows, prioritizing those most relevant to your use case.

| Recommendation                       | Description                                                                                                         | Implementation                                                                                                                                                                | Rationale                                                                                                                                                                                                                                                                                                                                               |
| :----------------------------------- | :------------------------------------------------------------------------------------------------------------------ | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Resource Protection & Limits**     | Implement rate limiting and connection limits to prevent resource exhaustion                                        | • **Rate Limiting**: Per-user, per-tool, or global rate limits with configurable thresholds • **Connection Limits**: Configurable concurrent client connection limits         | Prevents abuse, resource exhaustion, and protects server stability under high load                                                                                                                                                                                                                                                                      |
| **Security & Credential Management** | Apply least privilege principles and use short-term rotated credentials                                             | • Define minimal permission sets for each deployment mode • Implement automatic credential rotation with a limited time-to-live                                               | Reduces attack surface and limits exposure window if credentials are compromised                                                                                                                                                                                                                                                                        |
| **Configurable Tool Advertisement**  | Servers SHOULD support configuration-based control over which tools are advertised to clients                       | • Enable/disable specific tools via configuration • Support different toolsets per deployment context (dev/staging/prod)                                                      | Allows fine-grained control for different security contexts and testing scenarios                                                                                                                                                                                                                                                                       |
| **Documentation Search Tool**        | Servers SHOULD provide a tool to search relevant product documentation to provide contextual information to the LLM | • Search product docs, API references, guides, and knowledge bases • Return contextual snippets relevant to user queries • Enable LLM to retrieve domain-specific information | Allows LLM to access product knowledge and domain context without requiring all documentation in the system prompt                                                                                                                                                                                                                                      |
| **Metrics Emission**                 | Expose operational metrics (request counts, latencies, error rates)                                                 | Prometheus, OpenTelemetry or similar metrics export                                                                                                                           | Essential for monitoring, alerting, and performance optimization                                                                                                                                                                                                                                                                                        |
| **Context Overloading Prevention**   | Tools should be cautious about the amount of data returned to the output.                                           | For cases where the output can be too large, implement pagination, mandatory filtering, and let the client choose the response format (concise(default) vs. detailed)         | The more output is provided, the harder it will be for the LLM to deal with it. While there are some ways to mitigate this on the agent side, it’s always better to have this in mind already on the tool level.[more on Antrhopic's blog](https://www.anthropic.com/engineering/writing-tools-for-agents#returning-meaningful-context-from-your-tools) |

### MCP Servers with Code Execution Tools (EMERGING PATTERN)

**WARNING**: This usage pattern is emerging but **not yet recommended for production use** pending establishment of a reference architecture for sufficient sandboxing.

**Current Recommendation**: Prefer well-defined, bounded MCP tools over code execution tools. If code execution is required, implement it as a separate, highly-restricted opt-in capability with extensive security controls.

#### When to Use MCP Tools vs. Code Execution

| Approach           | Use When                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| :----------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **MCP Tools**      | • Access control is critical (fine-grained permissions, audit trails, compliance workflows) • Safety guardrails are non-negotiable (irreversible consequences requiring validation) • Abstraction reduces complexity (complex, inconsistent, or frequently changing APIs) • Performance matters (server-side caching, connection pooling, data co-location)                                                                                                     |
| **Code Execution** | • Multi-step workflow, filtering and transforming data, preventing intermediate data from influencing reasoning, parallel operations at scale • Flexibility is paramount (users need unpredictable operation composition) • APIs are well-designed (clean, documented APIs where abstraction adds little value) • Use cases are highly variable (pre-defining all tool combinations is impractical) • Rapid iteration is required (workflows change frequently) |

#### Requirements for Code Execution Tools (If Implemented)

| Requirement             | Description                                                                                       | Implementation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             | Rationale                                                                                                                                                   |
| :---------------------- | :------------------------------------------------------------------------------------------------ | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Explicit Allowlists** | Any code execution features MUST use explicit allowlists, never denylists                         | • Deny by default, allow explicitly • Allowlist permitted modules/packages • Allowlist permitted functions/methods • Allowlist permitted file system paths                                                                                                                                                                                                                                                                                                                                                                                 | Even with allowlists, code execution remains high-risk                                                                                                      |
| **Security Scanning**   | Servers accepting user-provided code or queries MUST implement security scanning before execution | • Static analysis for dangerous patterns (eval, exec, system calls) • Injection detection (SQL injection, command injection, path traversal) • Resource limit validation (memory, CPU, file size) • Deny dangerous operations before execution                                                                                                                                                                                                                                                                                             | Defense-in-depth; catch malicious or accidental dangerous operations. This is a mitigation, not a complete solution                                         |
| **Process Isolation**   | Code execution MUST run in a robust sandbox with strict security controls                         | • **Process isolation**: Separate process or container with resource limits • **File system**: Read-only access to approved paths only (or no file system access) • **Network**: Disabled by default; if required, allowlist specific endpoints • **Memory/CPU limits**: Configurable caps to prevent resource exhaustion • **Execution timeout**: Hard timeout limits (e.g., 30 seconds) • **Module allowlist**: Only approved libraries/packages available • **Privilege separation**: Run as unprivileged user with minimal permissions | Never trust user-provided code; assume malicious intent. Sandboxing is CRITICAL for preventing resource exhaustion, data exfiltration, and lateral movement |

**Recommended Sandbox Technologies**:

- **Linux containers with SELinux**: Traditional container isolation with mandatory access control policies (strong isolation)
- **Kata Containers**: Lightweight VM-based containers providing additional isolation layer (enhanced security for untrusted workloads)

**Additional Security Requirements**:

- Deploy in isolated environments (separate network segments, dedicated infrastructure)
- Implement extensive audit logging of all code execution attempts
- Establish incident response procedures for sandbox escape scenarios
- Regular security assessments and penetration testing required
- Implement rate limiting per user/session
- Test sandbox escape scenarios as part of security validation
